# Go profile

Catches stubs and dead code in Go projects (`go.mod` present). Uses `go vet` for static issues, then layers Go-specific greps for `panic("not implemented")`, empty receiver methods, and TODO clusters in handler files.

## Detection bash

```bash
# Go's built-in vet (looks for unreachable code, suspicious constructs)
go vet ./... 2> /tmp/stub-audit-govet.txt || true

# Explicit unimplemented panics in handlers / methods
rg -n \
  -e 'panic\(["\x27](not implemented|TODO|todo|unimplemented|stub|placeholder)' \
  --glob '*.go' --glob '!**/*_test.go' --glob '!**/vendor/**' \
  || true

# Empty method bodies: func (X *T) Foo(...) ... { } pattern
rg -n --multiline --multiline-dotall \
  -e 'func\s*\(\s*[a-zA-Z_][a-zA-Z0-9_]*\s+\*?[A-Z][A-Za-z0-9_]*\s*\)\s+[A-Z][A-Za-z0-9_]*\([^)]*\)[^{]*\{\s*\}' \
  --glob '*.go' --glob '!**/*_test.go' --glob '!**/vendor/**' \
  || true

# Empty top-level function bodies: func Foo(...) ... { } where Foo is exported
rg -n --multiline --multiline-dotall \
  -e 'func\s+[A-Z][A-Za-z0-9_]*\([^)]*\)[^{]*\{\s*\}' \
  --glob '*.go' --glob '!**/*_test.go' --glob '!**/vendor/**' \
  || true

# TODO/FIXME markers
rg -n -i \
  -e '//\s*(TODO|FIXME|XXX|HACK)' \
  --glob '*.go' --glob '!**/*_test.go' --glob '!**/vendor/**' \
  || true

# HTTP handlers returning hardcoded literal JSON with placeholder content
rg -n \
  -e 'json\.NewEncoder\([^)]*\)\.Encode\(.*?(mock|fake|sample|placeholder|TODO|todo)' \
  -e 'fmt\.Fprint(ln|f)?\([^,]+,\s*["`]\{[^}]*(mock|fake|sample|placeholder|TODO)' \
  --glob '*.go' --glob '!**/*_test.go' --glob '!**/vendor/**' \
  || true

# Stub-named declarations
rg -n \
  -e '\bvar\s+(mock|fake|dummy|sample|stub)[A-Z][A-Za-z0-9_]*' \
  --glob '*.go' --glob '!**/*_test.go' --glob '!**/vendor/**' \
  || true
```

Notes:

- `go vet` requires `go` on PATH and a buildable module. If the module does not build, vet returns errors before its checks. That is fine; the greps still run independently.
- Empty receiver methods are a strong Go-specific signal: a method on a concrete type with an empty body is almost always a stub awaiting implementation. Interface methods are declared without bodies, so this pattern only triggers on concrete implementations.

## Always-skip patterns (Go specific)

- Interface method declarations: `type X interface { Foo() }` has no body, do not flag.
- Empty struct types: `type Empty struct{}` is idiomatic for set semantics and signals.
- Mock implementations under `mocks/` or `*_mock.go` filenames generated by `mockgen` or `moq`.
- `init()` functions that legitimately have no body during early development (rare; mostly safe to ignore false positives here).
- Sentinel error variables: `var ErrNotImplemented = errors.New("not implemented")`. The declaration is not the lie, the call site is. Look for handler functions returning this sentinel and judge those, not the declaration.

## Graceful degradation

| Tool missing | Lost                                | Fallback                                                                              |
| ------------ | ----------------------------------- | ------------------------------------------------------------------------------------- |
| `go`         | `go vet` static analysis            | Greps still run and cover the high-value `panic("not implemented")` pattern.          |

Install Go from https://go.dev/dl/. The mechanical sweep produces useful candidates without `go` available, so a missing toolchain degrades coverage rather than blocking the audit.
